天网组织与人员总结

本总结将从以下四个方面呈现组织与人员模块从原型分析到代码开发的过程。

一、原型逻辑梳理与功能点提炼

（一）页面功能点

首 先，是拿到原型的一个思考过程。由原型可见，当点击导航栏的组织与人员时，页面上会展示组织架构，组织权限与人员权限三个模块。先分析点击的时候接口是如 何调用的，一个接口获取数据还是分多个。组织权限与人员权限依赖于组织架构，点击组织架构时会更新组织权限和人员权限，因此组织架构单独一个接口先请求。 组织权限与人员权限涉及增删改，我们希望增删改后只是局部数据更新，不要涉及其他模块，因此组织权限与人员权限分两个接口。这样就有了以下功能点：

1. 获取组织架构、
2. 获取组织下的物料组权限、
3. 获取相应组织的所有人员功能点。

其次，整理页面中其他显著地功能点：

4. 新增组织架构、
5. 更新组织架构、
6. 删除组织架构、
7. 配置物料组权限、
8. 删除物料组权限、
9. 新增人员
10. 修改人员
11. 删除人员

（二）弹窗功能点

最后，再提炼出弹窗里的相关功能点。
如下图，弹窗中的组织层级字段，输入组织代码后回车或失去焦点的时候匹配组织名称。这里采取用户输入一次请求一次的方式。

如下图，组织权限里的物料组权限另外加上了是否全选的辅助标志位，简化操作。物料组权限采用输入编码回车或失焦匹配的方式。

如下图，人员权限里的人员名称根据人员编码匹配，物料权限依赖于组织权限，同样根据物料编码匹配。

由此，弹窗里的功能点整理如下：

 12. 通过组织编码获取组织信息、
 13. 根据物料组编码获取物料组
 14. 根据邮箱号获取相应人员
 15. 获取组织下的物料编码对应的物料信息。

二、接口设计——接口名、请求方式、响应

整理出功能点再做接口的设计。一般增删改查采用post、delete、put、get的请求方式效率会更高一些。接口响应一般需要包括是否操作成功的标志位，数据对象，错误信息。 下图为组织与人员相关的接口。

三、组件拆分与搭建

组件拆分的依据：上图为组织与人员界面的组件结构图。秉承模块化思维和利于状态管理的原则将组件灵活拆分，实现组件资源的积累与复用。
（一）使用slot布局
由原型可知，层级与角色界面、组织与人员界面都为等比例的栅格布局，只是栅格中所填的组件不同。可将栅格布局先行抽离出来即组件AuthLayout，使用slot为组件预留位置。

<Row class="container">
    <Col :span="6" class="horizontal-section navigator-section">
         <slot name="navigator"/>
    </Col>
    <Col :span="18" class="horizontal-section">
         <slot/>
    </Col>
</Row>

在不同的页面按需引入组件即可。

  <SkynetAuthLayout>
    <SkynetOrganizationPanel slot="navigator" :organizationId="organizationId"  @selectNode="getHierachy"/>
    <SkynetOrgAndUserMemu :organizationNode="organization"/>
</SkynetAuthLayout>

（二）弹窗的引入
一般地，弹窗的modal和页面中操作弹窗的按钮放在同一页面会简化数据传递的过程，不用为了v-modal、v-title、loading等属性再做状态管理。而弹窗里的内容可单独抽离成组件，如editOrgMode组件，组件里为form表单。

这 里特别为editOrgMode组件加上v-if属性，在v-if生效后，editOrgMode组件才会渲染props里的数据，而如果没有v-if属 性的控制，在未操作的情况下由于v-modal的原因，弹窗虽然不会出现，但实际上它已经在试图渲染props的数据，在拿不到数据的情况下就会报 TypeError的错误。

    <Modal
      v-model="isshoweditorgmode"
      :loading="modalLoading"
      :title="modeltitle"
      @on-ok="submit">
        <editOrgMode
          v-if="isshoweditorgmode"
          ref="organizationEditForm"
          :organization="organization"
          :materialGroupsInfo="materialGroupsInfo"
          @submitSuccess="handleUpdateOrganizationSuccess"
          @submitFailed="handleUpdateOrganizationFailed"
        />
    </Modal>

（三）computed属性的应用
(1)弹窗标题
当编辑和新增的弹窗字段相同时，可以给弹窗绑定title属性，其值在computed里计算。

  modeltitle() {
     return OPERATION_USER_MODAL_TITLE[this.operateType];
   },

当不同的按钮被点击时，给予operateType不同的值,OPERATION_USER_MODAL_TITLE数组的值也随之更新。

this.operateType = OPERATION_USER_OPTIONS.ADD;

(2)绑定文本值
当页面上要渲染的内容除了数据以外还有其他内容时，为了美化页面，让页面在无数据时不显示任何内容。可将要渲染的内容放在computed里计算，于此同时，为元素绑定v-if的属性，用绑定title的方法建立元素和computed的联系。

<div v-if="organizationPathText" :title="organizationPathText">{{ organizationPathText }}
    </div>
   organizationPathText() {
       if (this.organization && this.organization.getPath()) {
       return `所属组织:${this.organization.getPath()}`;
  }
  return "";
}

（四）按钮组的引入
我们希望以更小模块组合的方式编写页面，这样可以提供组件的复用率，更多地从vue模块化的思维中受益。因此，我们将组件拆分到按钮组的力度。以下为按钮组的代码，包括了编辑和删除按钮：

   <Icon type="skynet-delete" class="user-operation-button delete-icon" @click.native="onDeleteClick"/>
    <Icon type="skynet-edit" class="user-operation-button edit-icon" @click.native="onEditClick"/>

在按钮组件的点击事件中，只将点击事件的触发提交到父组件，具体点击后的操作业务由父组件完成，将组件和业务剥离开可以提高组件的复用率。

onDeleteClick() {
this.emit('deleteUser');
    },
    onEditClick() {
      this.emit('operateUser', { type: OPERATION_USER_OPTIONS.EDIT });
},

这里编辑事件提交时加上了type参数，在父组件的同名 operateUser事件对应的函数中，会判断type的类型，根据操作类型做不同的业务。type的值还会赋值给第三部分computed应用里提到 operateType，这样弹窗就可以根据点击的事件显示不同的标题。

四、数据仓库与请求层搭建

（一）数据仓库

通 过第一部分功能点分析可知，组织架构，组织权限，人员权限在获取信息后都会随着用户的操作进行更新，并且组织权限和人员权限依赖于组织架构，因此需要用到 状态管理来实现数据的共享和更新。上图为状态管理分析图，将状态管理分为两个模块，涉及三个获取信息的接口，在getters里将要管理的数据按需处理， 如增删字段或格式化数据等操作。

（二）请求层

上图为请求层的结构图，对于不涉及更新与共享的的接口，都可按模块放在请求层里。组织与人员界面分为用户模块，组织相关模块，物料组模块，并且对axios进行封装，简化请求层的代码。

  export const editOrganization = (id, code) =>
    ajax.put(baseUrl + `/organizations/${id}`, { code });

（三）状态管理的流程

下图为组织架构的状态管理，包括了数据获取到应用的过程，以及父子组件交互。

（1）请求层在获取数据后，将数据调教到mutation，这样state就可以获取到请求的数据，可以按照具体的需求决定是否将state里的数据在getters里进行处理，如字段的增删，数据格式化等处理。这里处理好的数据变量为organizations.
（2）在父组件里获取处理好的数据，首先引入mapaction这样才可以调用请求，再引入mapgetters获取数据；然后封装调用请求的函数，这样 父子组件可以共同调用这个请求，子组件不需要再引入mapaction。父组件将获取的organization和封装的函数loadData方法通过绑 定的方法传递给子组件1。这里的子组件1为渲染组织架构的树形结构。
（3）子组件使用props接受父组件传递的函数和数据，并在创建界面时调用。这样树形结构会在页面加载后渲染完成。
（4）树形结构渲染完成后可对树进行操作，当选择树的不同节点时，需要将获取的节点信息及时更新给子组件。使用emit方法将获取的节点提交到父组件。
（5）emit会触发父组件绑定的同名方法，在这个方法里，父组件可以接受到子组件更新过来的数据。在实际项目中，组织权限和人员权限依赖节点信息请求数据。因此，此处父组件需要将节点信息传递给子组件，这里的子组件为组织权限和人员权限模块。
（6）子组件在props里接受数据后并应用。
由此，便完成了从获取数据，子组件向父组件传递数据，父组件向子组件分发数据的过程。
